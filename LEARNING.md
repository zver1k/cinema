# Учебный конспект — проект Cinema

Журнал изучения Next.js / React / TypeScript на пет-проекте «кинотеатр».
Цель файла — портативная память: переносится между компьютерами через git.

## Стек
- **Next.js** (App Router; нестандартная сборка — гайды в `node_modules/next/dist/docs/`, читать перед написанием кода)
- **Prisma + PostgreSQL**, **Better Auth**, **Tailwind**, **shadcn/ui**, **Swiper**, **MagicUI**
- Данные о фильмах — **Kinopoisk API Unofficial** (`kinopoiskapiunofficial.tech`), OpenAPI-спека: https://kinopoiskapiunofficial.tech/documentation/api/openapi.json
  - ключ в `API_KEY`, базовый URL в `API_URL`; загрузчики в `src/shared/api/*.ts`, путь `…/api/v2.2/films/{id}/…`

## Команды / окружение
- Prisma — через `bun --bun run prisma ...` (грузит `.env`; `npx` не подхватывает `DATABASE_URL`)
- CLI `tsc` НЕ работает (tsconfig ждёт TS 6.0) → проверка типов только в Webstorm; базовая проверка — `npm run lint`
- ⚠️ **ESLint НЕ ловит**: несуществующие не-JSX идентификаторы (`view`, `emptyStates`), хотя `react/jsx-no-undef` ловит ненайденные JSX-компоненты. Типы краснеют только в Webstorm.
- Dev-сервер обычно запущен — результат виден сразу.

## Что построено
- **Auth**: логин/регистрация, защита `/profile`, восстановление пароля.
- **Избранное / Watchlist**: модели `Favorite`/`Watchlist` (+ enum `WatchStatus`), server actions `toggleFavorite`/`setWatchStatus`, кнопки на странице фильма с **оптимистичным откликом** (`useOptimistic`, клиентские `FavoriteButton`/`WatchButtons`).
- **Профиль**: вкладки (избранное/просмотрено/планы/настройки), `MovieGrid`, счётчики, сортировка (`?sort` по имени/году/рейтингу), переключатель Сетка/Список (клиентский `useState`).
- **Страница фильма**: постер, действия, бокс-офис («Бюджет и сборы»), «Факты и ляпы» (2 случайных факта), актёры (с кнопкой «Показать всех»), режиссёры, съёмочная группа (группировка в Аккордеоне), рецензии, «где смотреть», похожие.
- **Лента `/movies`**: страница-список — серверный `async`-компонент фетчит `getFilms()` и рисует `MovieGrid` (грид). **Пагинация через URL** (`?page=`): кнопки «Назад/Вперёд» на `<Link>`, на краях — `disabled` без ссылки.
- **UI-эффекты**: SparklesText (логотип), AuroraText (заголовки подборок).

## Концепции (главное, что усвоено)

### Server / Client компоненты (RSC)
- Серверный компонент: `async`, фетчит данные, НЕ имеет `useState`/`onClick`/`useEffect`.
- Клиентский: первая строка `"use client"`; только там живут хуки и обработчики событий.
- Паттерн: **сервер фетчит данные → передаёт пропсами → клиент рисует интерактив** (`ProfileMovies`, `CastList`).
- **Граница server→client**: пропсами едут только СЕРИАЛИЗУЕМЫЕ данные. Фильтрация и тяжелая логика (группировка) — на СЕРВЕРЕ; клиент получает готовый чистый массив.

### Трансформация данных и Группировка
- **Метод `.reduce()`**: мощный инструмент для преобразования массива в объект (Record). Позволяет за один проход разложить элементы «по коробкам» (например, по профессиям).
- Синтаксис: `array.reduce((acc, item) => { ... return acc }, {} as T)`. Важно всегда возвращать аккумулятор и инициализировать его правильным типом.
- **Data-driven группировка**: использование внешнего справочника (`PROFESSION_LABELS`) как фильтра и источника названий одновременно.

### Компонентный подход (DRY)
- Вынос повторяющейся верстки в мелкие компоненты (`CastItem`).
- Если компонент не использует хуки, его можно оставить серверным (RSC) для лучшей производительности.
- Передача объекта целиком через проп (`member: Staff`) делает интерфейс чище, но требует внимательной типизации аргументов функции: `function Comp({ member }: Props)`.
- **Переиспользуемый компонент владеет только ОБЩИМ.** То, что различается между местами использования (контейнер/размеры, `sizes`, `loading`), — отдать пропсами или родителю, не зашивать чужую вёрстку. Пример: `PosterImage` рисует только `<Image>` + фолбэк, а `relative`-контейнер с размером/скруглением даёт родитель (карточка грида, строка, страница фильма — у всех разный).
- При выносе логики дублирующий код «худеет»: `Poster` был клиентским (`useState`+`onError`), после выноса в `PosterImage` стал тонкой серверной обёрткой.

### Картинки и фолбэк
- **Фолбэк битой картинки**: `<Image onError={() => setError(true)}>` → при ошибке прячем картинку (`hidden`) и показываем заглушку. `onError` — обработчик события → компонент должен быть клиентским (`"use client"`).
- **`Skeleton`/`animate-pulse` = ВРЕМЕННОЕ ожидание** (грузится). Для «данных нет / не загрузилось НАВСЕГДА» — статичный плейсхолдер (`bg-muted` без пульсации + иконка `ImageOff` по центру). Пульсирующий скелетон на постоянную ошибку обманывает — выглядит как «висит».
- Заглушка поверх `<Image fill>` сама должна быть `absolute inset-0` (картинка `fill` — абсолютная; обычный flow-div схлопнется в 0 высоты).

### Оптимистичный UI (`useOptimistic`)
- **Зачем**: показать ожидаемый результат МГНОВЕННО, не дожидаясь ответа сервера. При ошибке React откатывает состояние сам.
- **Где живёт**: это React-хук → только в клиентском компоненте (`"use client"`). Серверная страница фетчит начальное состояние и отдаёт пропсом — паттерн «сервер фетчит → клиент рисует».
- **Сигнатура**: `const [optimistic, setOptimistic] = useOptimistic(actualState, (current, next) => newState)`. `actualState` — пропс с сервера (источник правды); кнопку рисуем от `optimistic`, не от пропса напрямую.
- **`action` формы становится inline async**: сначала `setOptimistic(...)` (предсказание), потом `await serverAction(...)` (реальная мутация + `revalidatePath`). `.bind(null, ...)` тут НЕ годится — нужно два действия по очереди.
- **Редьюсер обязан зеркалить серверную логику**, иначе UI «дёрнется» при ответе. Булево → `(_, next) => next`; enum-тогл (как watchlist) → `(cur, clk) => cur === clk ? null : clk` (ровно как `if (existing.status === status) delete else upsert` в экшене).
- **Общее состояние — общий хук**: две кнопки watchlist меняют один статус → один `useOptimistic` на обе, не два независимых (иначе рассинхрон подсветки).
- **Когда оправдан**: частые, лёгкие, легко-откатываемые действия (лайк, избранное, чекбокс). НЕ для необратимого/важного (оплата, удаление) — там честно ждать сервер.
- Минус: без JS не работает (серверная `<form action>` работала бы и без JS — progressive enhancement). Для лёгких тоглов компромисс оправдан.

### TypeScript
- **`Partial<Record<Keys, Type>>`**: когда объект содержит только часть ключей из перечисления (например, не все профессии из API нам нужны в справочнике).
- **Дженерики `useOptimistic<State, Action>`**: явно типизировать, когда тип «что показываем» (`WatchStatus | null`) отличается от типа «что передаём в set» (`WatchStatus`).
- **Структурная типизация — проси самый узкий тип, который используешь.** Компонент не должен требовать «жирный» конкретный тип, если читает горстку полей. Описать минимальный тип пропа (`MovieItem`) — и ему подойдут любые объекты с нужными полями (`Film` И `FilmDetail`). Так `MovieGrid` стал переиспользуемым для каталога и профиля сразу.
- **`?` vs `| null` — разное!** `field?: T` — ключа может вообще не быть. `field: T | null` — ключ обязан быть, но значение может быть `null`. Чтобы тип подошёл объекту, где поля НЕТ совсем (`Film` без `filmLength`), нужен именно `?`.

### React / JSX
- Правая часть `&&` = ОДИН узел; несколько детей → один `<>...</>` (Fragment); JS-выражение в разметке → в `{ }`.
- `key` — на самом внешнем элементе, который возвращает `.map`. Для key + несколько соседей: `<Fragment key={i}>` (короткий `<>` не принимает key).
- Сепаратор между элементами: `{i < items.length - 1 && <Separator/>}`.
- Тернарник `активна ? "secondary" : "ghost"` — подсветка всегда в ПЕРВОЙ ветке (легко перепутать).
- `dangerouslySetInnerHTML={{ __html }}` — для рендера сырого HTML (React по умолчанию экранирует `{text}`); у такого элемента не может быть детей; контейнер — `<div>` (разрешает блочное).
- Data-driven рендеринг: справочник-объект/массив (`emptyStates`, `sorts`, `typeLabels`) вместо копипаста разметки; `Record<...>` + `.map`.

### Next.js: загрузка и стриминг (`loading.tsx`)
- **`loading.tsx`** рядом с `page.tsx` — конвенция: Next САМ оборачивает страницу в `<Suspense fallback={<Loading/>}>`. Пока `async`-страница ждёт данные — виден фолбэк, потом контент подменяется (стриминг). Ручной `<Suspense>` писать не нужно.
- Серверный компонент, БЕЗ пропсов, **default export**. Оборачивает `page.tsx`, но не `layout.tsx` того же сегмента.
- **Скелетон повторяет сетку контента** (те же `grid-cols`, `gap`, `aspect-2/3`) — иначе вёрстка прыгнет при подмене.
- N заглушек без данных: `[...Array(20)].map((_, i) => <Skeleton key={i} .../>)`. Готовый `Skeleton` = `animate-pulse rounded-xl bg-muted`.
- Проверка стриминга в HTML: в одном ответе есть И фолбэк-скелетон, И реальные карточки. Увидеть глазами — троттлинг Slow 3G в DevTools.

### Next.js: URL как состояние (`searchParams`)
- **`searchParams` — это `Promise`** (Next 16): `async function Page({ searchParams }: { searchParams: Promise<{ page?: string }> })` → `const { page } = await searchParams`. Причина: статику Next рендерит наперёд, а query известно только в момент запроса; `await` осознанно переводит кусок в динамический рендер. Так же устроен `params` для `[id]`.
- **`params` ≠ `searchParams`**: `params` — сегменты пути (`[id]`), `searchParams` — query после `?`. У страницы без `[...]` сегмента `params` пустой.
- **Из URL всё приходит строкой** (`?page=3` → `"3"`). В число: `Number(page) || 1`. Порядок важен — `|| 1` ловит `NaN` ПОСЛЕ `Number` (и для пустого, и для мусора `?page=abc`); `Number(page ?? 1)` мусор не ловит (`??` срабатывает только на null/undefined, а `"abc"` для него «есть»).
- **URL-пагинация = `<Link href="?page=...">`** (не кнопки с onClick). Клик меняет `searchParams` → страница перефетчивается. Без клиентского JS.
- **«Выглядит disabled» ≠ «заблокировано»**: `disabled` на кнопке внутри `<a>` — косметика, ссылка всё равно живая. Чтобы реально не листало на краю — условный рендер: `isFirst ? <Button disabled> : <Button asChild><Link/></Button>` (на краю ссылки нет совсем).
- **`asChild`** (shadcn/Radix `Slot`): `Button` не рисует свой `<button>`, а «переливает» стили на дочерний `<Link>` → один валидный `<a class="кнопка">` вместо невалидной вложенности `<a><button>`. Приём для любого «кнопочного вида на ссылке».

### CSS / Tailwind
- `gap`/`flex` ставится на РОДИТЕЛЯ (разносит детей), не на каждого ребёнка.
- `<Image fill>` требует родителя с заданным размером (высотой): `aspect-2/3` или `h-28 w-20`.
- Скругление картинки: `rounded-*` + `overflow-hidden` на контейнере (один `border-radius` не обрезает содержимое).
- `toLocaleString("ru-RU")` для разрядов чисел (`25000000` → `25 000 000`).

### Next.js: `generateMetadata`
- Экспортируемая `async` функция рядом с `page.tsx` — Next вызывает её сам для `<title>` и OG-тегов.
- Получает те же `params`/`searchParams` что и страница (тоже `Promise` — нужен `await`).
- Используй `getFilmByIdSafe` (возвращает `null`) вместо `getFilmsById` (вызывает `notFound()`) — иначе крашится при несуществующем id.
- При `null` возвращай `{}` — Next подставит дефолтные теги.

### Несколько URL-параметров одновременно
- `new URLSearchParams(searchParams.toString())` — копирует текущие params; меняй только нужные, остальные сохранятся.
- При смене жанра/типа — сбрасывай `page` (`params.delete("page")`), иначе пользователь попадёт на несуществующую страницу.
- Пагинация должна сохранять ВСЕ активные фильтры (`keyword`, `type`, `genre`) — строй href через хелпер `paginationParams(p)`.

### Нормализация данных из разных эндпоинтов
- Разные эндпоинты API возвращают разные поля (`filmId` vs `kinopoiskId`, `rating: string` vs `ratingKinopoisk: number`).
- Нормализуй в загрузчике (`.map(f => ({ kinopoiskId: f.filmId, ... }))`), чтобы компоненты работали с единым типом (`MovieItem`).
- `parseFloat(f.rating) || null` — конвертация строки рейтинга в число с защитой от `NaN`.

### Debounce через `useEffect`
- `setTimeout` + `clearTimeout` в cleanup: пока пользователь печатает — таймер сбрасывается, запрос уходит только после паузы (~400ms).
- Пустое значение — отдельный кейс: на целевой странице сбрасывай URL (`router.push("/movies")`), на других — ничего не делай.

### Переиспользование страниц (пример: `/series`)

- Если новая страница отличается от существующей только одним параметром — не дублируй логику, параметризуй API-функцию.
- `getFilms` принял объект `{ genreID?, page?, type? }` вместо позиционных аргументов — теперь `/series` просто передаёт `type: "TV_SERIES"`, всё остальное то же.
- **Объект параметров vs позиционные аргументы**: при 3+ опциональных параметрах объект предпочтительнее — порядок не важен, легче расширять, читается как именованные параметры. Паттерн такой же как у `searchParams: Promise<{...}>` в Next.js.
- `activeGenre || undefined`: значение `0` («не выбран») надо явно превращать в `undefined` перед передачей в функцию, иначе `if (genreID)` не спасёт — `0` уже попадёт в URLSearchParams.
- **Где лежит компонент — вопрос принадлежности**: `GenreFilter` в `movies/_components/` работал, но архитектурно неверен — его использовали два разных роута. Компонент переехал в `shared/ui/` — теперь принадлежит всем.
- Фиксированный `type` (сериалы всегда `TV_SERIES`) не нужно класть в `searchParams` страницы — он не меняется пользователем. Передай константой прямо в вызов API.
- Имя компонента-страницы должно совпадать с роутом (`SeriesPage`, не `MoviesPage`) — важно для дебага в React DevTools и читаемости стектрейсов.
- **`generateMetadata` требует явного возвращаемого типа**: `Promise<Metadata>` из `"next"`. Без аннотации Next.js TS-плагин выдаёт TS71008. Импортировать лучше как `import type { Metadata }`.

### Устойчивость API-функций к 404

- Разные эндпоинты ведут себя по-разному при 404: одни возвращают «объект не найден», другие просто не имеют данных для конкретного тайтла (актёры, бюджет, факты у сериалов).
- **Функции возвращающие массив** (`getCastById`, `getBoxOffice`, `getFilmFacts`) при 404 должны возвращать `[]` — страница рендерится, секция просто пуста.
- **Функции возвращающие объект** (`getFilmsById`) при 404 вызывают `notFound()` — показывать страницу нечего.
- Паттерн: `if (response.status === 404) return []` перед `if (!response.ok) throw` — сначала обработать «ожидаемое отсутствие», потом непредвиденные ошибки.
- Проверять нужно все функции которые используются на новых страницах — то что работает для фильмов может падать для сериалов из-за неполноты данных в API.

### Страница персоны (`/person/[id]`) и связка страниц

- **Один эндпоинт — много сущностей**: `/api/v1/staff/{id}` отдаёт *любого* человека (актёр, режиссёр, композитор). Поэтому загрузчик — `getPersonById`, а не `getActorById` (имя не должно врать). Кладём рядом со `staff`-логикой (`cast.ts`).
- **404 для загрузчика-объекта** → `notFound()` (как `getFilmsById`): нет человека — нечего показывать. `[]`/`null` — для массивов / `generateMetadata`.
- **Связка «клик → страница»**: чтобы навигация заработала, недостаточно создать роут — нужно *проставить ссылки*. `CastItem` обернули в `<Link href={`/person/${member.staffId}`}>`. ⚠️ id персоны в типе `Staff` зовётся `staffId` (не `personId`, не `kinopoiskId`) — то же значение, другое имя в этом эндпоинте.

### Предел структурной типизации — когда переиспользовать НЕЛЬЗЯ
- `MovieGrid` ждёт `MovieItem` (с `posterUrl`, `genres`, `year`), а `person.films` (`PersonFilm`) этих полей **физически не содержит** — API их не отдаёт. Структурная типизация спасает, только если данные *есть*; нельзя сузить к полю, которого нет.
- Достать постеры = N отдельных запросов на каждый фильм → дорого. Вывод: фильмографию рисуем **текстовым списком**, а не сеткой постеров. Не каждый список фильмов = `MovieGrid`.
- Плюс переименования полей: `filmId` (не `kinopoiskId`), `rating: string` (не `ratingKinopoisk: number`) — и имя, и тип другие.

### Тип — это контракт с НАШЕЙ стороны (API его не обязан соблюдать)
- `PERSON_ROLE_LABELS: Record<StaffType, string>` обещал «для любого ключа — строка». Но в данных прилетел `professionKey` вне нашего enum → лукап вернул `undefined`, роль отрисовалась пустой. Тип описывает *наши ожидания*, рантайм-данные могут им противоречить → защита (`|| ""`, `&&`) всё равно нужна.
- **`Record<K,V>` vs `Partial<Record<K,V>>`**: полный `Record` заставляет заполнить ВСЕ ключи (TS-ошибка иначе), зато лукап «всегда строка» без `undefined`-проверок. `Partial` — можно покрыть часть, но лукап возвращает `T | undefined`. Сделка: полнота-обязанность ↔ безопасность-без-проверок. Разные справочники под разные задачи (`PROFESSION_LABELS` — группировка съёмочной группы, мн.ч., без `ACTOR`; `PERSON_ROLE_LABELS` — роль в фильме, ед.ч., с `ACTOR`).

### `key` и реальные дубли
- Один человек может числиться в **одном фильме несколько раз** (Нолан в «Мементо» — Режиссёр + Сценарист) → `filmId` в одиночку **не уникален**, как `key` не годится.
- Для **статичного** серверного списка (не переупорядочивается, не мутируется) `key={i}` допустим. Идеал — **составной ключ** `${filmId}-${professionKey}`: и смысловой, и уникальный при двойной роли.

### Иммутабельные методы массива
- `.filter()` / `.map()` / `.slice()` / `.reduce()` возвращают **новый** массив, исходный не трогают. Вызов `info.filter(...)` без присвоения = no-op (результат в никуда). Надо `const visible = info.filter(...)` и мапить по нему (или зачейнить `.filter().map()`).
- **Data-driven инфо-блок**: массив `[{label, value}]` → `.filter(r => r.value)` (пустые `null`/`""` отваливаются сами) → `.map` в `<li>`. Один `<ul>` снаружи, `<li>` внутри `.map` (а не `<ul>` на каждый пункт).

### Группировка фильмографии (повтор `.reduce()`)
- Фильмографию персоны сгруппировали по `professionKey` тем же `.reduce()`, что и съёмочную группу (`cast-section.tsx`): `{} as Record<string, PersonFilm[]>`, `Object.entries(grouped).map()`. Группировка решает дубли (один фильм под несколькими ролями) и читается как на взрослых сайтах. В заголовке группы — подпись роли + счётчик.
- ⚠️ **JSX-ловушка**: `{LABELS[key](count)}` — JS читает как **вызов** строки как функции → рантайм-краш `is not a function`. Надо `{LABELS[key]} ({count})` — скобки тут текст-литерал, число в своих `{}`.
- `Record<string,T>`-аккумулятор → ключ группы имеет тип `string`, а справочник ждёт `StaffType` → нужен `key as StaffType` при лукапе.

### Tabs (Radix) поверх группировки + RSC-нюанс
- **Серверный компонент МОЖЕТ рендерить клиентский** (наоборот — нельзя). `tabs.tsx` имеет `"use client"`, но `Filmography` остаётся серверным: он просто рендерит `<Tabs>`, а списки фильмов едут как `children` (отрендерены на сервере). `"use client"` на `Filmography` НЕ нужен.
- Табы динамические → генерируем из `Object.entries(grouped)`: `.map` дважды (триггеры в `TabsList`, контенты ниже). **`value` у `TabsTrigger` и `TabsContent` обязан совпадать** (ключ группы) — иначе клик по вкладке ничего не покажет.
- `defaultValue` = первый ключ (`entries[0]?.[0]` / `Object.keys(grouped)[0]`), т.к. роли заранее неизвестны.
- ⚠️ Не путать «все группы» и «первый ключ»: `Object.entries(grouped)` (весь массив для `.map`) ≠ `...[0][0]` (один ключ для `defaultValue`). Один `[0]` не туда — и `.map` мапит по символам строки.
- Подпись вкладки — **инлайн-текст**, не блочный `<h3>` (`TabsTrigger` = `<button>`, заголовок внутри кнопки невалиден).

### `as` врёт компилятору + реальные данные грязнее типа (продолжение урока)
- Фильмография прислала `professionKey` ВНЕ нашего `StaffType`: `HIMSELF`/`HERSELF` (играет себя — камео/документалки), `HRONO_TITR_MALE/FEMALE` (хроника). `key as StaffType` заглушил TS → лукап вернул `undefined` → **пустая подпись вкладки**. `as` = «поверь мне», а не проверка.
- Чинили двусторонне: **расширили `StaffType`** реальными значениями + подписи в `PERSON_ROLE_LABELS` (отразить известное), **и** страховочный фолбэк `?? "Прочее"` на лукапе (поймать будущие сюрпризы API). Урок: тип под известное + soft-fallback под неизвестное.
- **Дедуп от API-дублей**: API кладёт один `filmId` в группе по 2–3 раза (актриса играла нескольких персонажей / редундантность). Чистим в `.reduce()` перед пушем: `if (!acc[key].some((f) => f.filmId === film.filmId)) acc[key].push(film)`. `.some()` — булево «есть хоть один?», без лишнего массива (в отличие от `.filter().length`).

### Ловушка авто-импорта Webstorm
- При вводе `string` IDE молча добавила `import { string } from "zod"` — линт ловит как unused, но легко проглядеть. После кодинга проверять список импортов.

## TanStack Query (React Query) — клиентский фетчинг и кэш

### Когда он вообще нужен в RSC-приложении
- В нашем стеке (**RSC + server actions**) почти всё уже закрыто: загрузка → серверные `async`-компоненты, мутации → server actions + `useOptimistic`, loading → `loading.tsx`/Suspense. **Строго говоря, React Query не обязателен** — проект без него не сломается.
- Он нужен **ровно там, где данные грузятся на КЛИЕНТЕ, повторно, по действию пользователя**, а сервер в этой цепочке не участвует: автокомплит-поиск, бесконечная прокрутка, поллинг. Там же, где серверный фетч — он лишний (медленнее, без SEO).
- **Не переписывай рабочий URL-поиск на клиент**: страничный поиск через `router.push("/movies?keyword=")` + серверный рендер — это shareable URL + SEO + кэш Next. Перевод на клиент = шаг назад. React Query сюда добавляют как ОТДЕЛЬНУЮ фичу (выпадашка-автокомплит), не вместо.

### Два разных кэша (Next vs React Query) — не путать
| | Кэш Next.js (Data Cache) | Кэш React Query |
|---|---|---|
| Где | сервер | браузер (RAM) |
| Для кого | все юзеры | один юзер/вкладка |
| Что кэширует | серверные `fetch` (RSC), напр. `next: { revalidate: 3600 }` | клиентские запросы по `queryKey` |
| Переживает reload | да | нет (по умолчанию) |
- Они **не пересекаются**. Как только фетч идёт с клиента — Data Cache Next в цепочке нет вообще.

### Граница «клиент не может фетчить напрямую» (секрет в env)
- Загрузчики (`getFilms`, `getFilmsByKeyword`) читают `process.env.API_KEY` — **серверный секрет**. Импортнуть их в клиентский компонент = утечь ключ в браузерный бандл.
- Мост браузер→сервер для `queryFn` — **Server Action** (тонкая обёртка `"use server"` вокруг существующего загрузчика). Предпочтительнее Route Handler: сквозная типизация, меньше boilerplate, единообразно с нашими мутациями. Route Handler — когда API нужен ВНЕШНИМ потребителям.
- Файл-обёртка живёт рядом с виджетом-владельцем (`widgets/Search/actions.ts`), не привязан к странице.
- ⚠️ В файле `"use server"` **каждая экспортируемая функция обязана быть `async`** (server action под капотом — сетевой вызов). `"use server"` ≠ `"use client"` — они противоположны (server = «остаётся на сервере», client = «едет в браузер»).
- ⚠️ Грабля рекурсии: внутри `searchMovies` написать `return searchMovies(...)` — бесконечная рекурсия (звал себя вместо `getFilmsByKeyword`). Правило: `return X(...)` внутри функции `X` почти всегда баг.

### `useQuery` — три кита
- `queryKey: ["search", value]` — **удостоверение данных в кэше И список зависимостей**. Меняется ключ → летит новый запрос. Заменяет `useEffect([value])` декларативно.
- `queryFn: () => searchMovies(...)` — **функция-обёртка, не вызов**. RQ должен звать её САМ, когда решит, а не в момент рендера. `queryFn: searchMovies(...)` (с вызовом) — ошибка.
- `enabled: value.length > 0` — пока `false`, хук «спит», `queryFn` не вызывается (не слать запрос на пустую строку).
- Возвращает синхронно `{ data, isLoading, isError }`: `data` = `undefined` пока грузится; когда придёт — RQ триггерит ре-рендер. ~4 строки вместо ручных `useState`×3 + `useEffect` + отмена гонок.

### Ловушка `async`: клиентский компонент НИКОГДА не async
- «Я гружу данные → надо `async`» — неверно. `async function Component` бывает только у **серверных** компонентов. Клиентский + `async` = ошибка `<X> is an async Client Component`.
- `useQuery` **не возвращает промис** — он прячет асинхронность внутри и общается через **состояние во времени** (ре-рендер), а не через `await`. Компоненту не надо «ждать» — он реагирует на смену стейта.

### Дебаунс запроса (отдельный стейт)
- `queryKey: ["search", value]` дёргается на каждую букву → запрос на каждую букву. Чтобы дебаунсить — завести второй стейт `debouncedValue`, в `queryKey`/`queryFn`/`enabled` использовать его, а `setDebouncedValue(value)` вызывать в `setTimeout` (~400ms) внутри `useEffect([value])`.
- **`useEffect` — одна ответственность**: после рефактора в нём ТОЛЬКО `setDebouncedValue`. Навигацию (`router.push`) убрать оттуда — она теперь явное действие (кнопка «Поиск»/Enter), а не авто-сайд-эффект набора. Два параллельных пути к данным (useQuery + router.push) — запах, оставить один.
- **Урок про дебаунс**: при быстром наборе промежуточные значения вообще НЕ становятся `queryKey` (таймер сбрасывается). Поэтому «промах кэша» на пути вниз — часто не промах, а ключ, которого в кэше и не было.

### `staleTime` vs `gcTime` (кэш на практике)
- По умолчанию `staleTime: 0` → данные «протухают» мгновенно: RQ отдаёт из кэша (UI мгновенный), НО **в фоне всё равно перезапрашивает**. Для поиска лишнее.
- `staleTime: 1000*60*5` → в пределах 5 мин данные «свежие» → отдать из кэша **без запроса к серверу**. Вот тут «набрал бэт, стёр, снова бэт → мгновенно».
- `gcTime` (дефолт 5 мин) — другое: сколько держать НЕиспользуемые данные перед удалением из памяти. `staleTime` = «надо ли рефетчить», `gcTime` = «когда выкинуть из RAM».
- **Доказательство кэша в логах**: на пути вниз ключи, уже виденные вверху (`Друзь`, `Друз`), `searchMovies` НЕ дёрнули (из кэша), а новые (`Дру`, `Др`) — дёрнули. Видно глазами по отсутствию POST на закэшированные ключи.

### Грабли стека (React 19 + React Compiler + RQ)
- ⚠️ **«A component was suspended by an uncached promise»**: результат `useQuery` v5 имеет скрытый геттер `.promise` (задел под `use()` из React 19). `reactCompiler: true` при мемоизации **читает свойства объекта** → задевает `.promise` → создаёт промис В РЕНДЕРЕ → Suspense зацикливается (спам в логах).
- Фикс — директива **`"use no memo";`** первой строкой в теле компонента (штатная «дверь выхода» React Compiler, отключает компиляцию для одного компонента). Вывести логикой нельзя — чистое знание окружения.
- Это **класс проблем**: тот же React Compiler ругается на `watch()` из react-hook-form («Compilation Skipped: incompatible library»). На bleeding-edge стеке баг бывает не в твоём коде, а на стыке инструментов.

### Лимит внешнего API — почему дебаунс/кэш не косметика
- ⚠️ Неофициальный Кинопоиск API: лимит ~**500 запросов/сутки**, при превышении — **HTTP 402**. Наши загрузчики на 402 молча возвращают `{ items: [] }` → симптом «Найдено: 0 всегда» (код исправен, API заблокировал).
- Запрос-на-букву + параллельный `router.push` сожгли суточный лимит за полчаса тестов. На API с жёстким лимитом дебаунс + `staleTime`-кэш — это разница между «работает» и «402 до завтра», а не оптимизация на потом.
- Проверка статуса API напрямую: `curl -s -o /dev/null -w "%{http_code}" -H "X-API-KEY: $KEY" "$URL/..."`.

### Проверь, прежде чем усложнять (роуты фильм/сериал)
- Инстинкт «тип результата разный (фильм/сериал) → роуты разные → ветвить href» здравый, НО сначала проверь, есть ли разница. У нас `/movies/[id]` и `/series/[id]` **идентичны** (один `getFilmsById`, те же компоненты) → ветвление было бы кодом ради несуществующей разницы. Все результаты ведём на `/movies/[id]`.
- (Дубль `series/[id]` — запах, две одинаковые страницы стоит слить. Отдельная задача.)

## Кэширование и экономия квоты внешнего API (реальный кейс)

### Сначала измерь, потом чини
- Симптом «402, квота выжжена» → первая догадка была «билд жжёт запросы». **Дашборд Vercel (Observability) показал другое**: `/movies/[id]` — 2.3K инвокаций за 12ч, `/person/[id]` — 1.5K. Виноват **рантайм-трафик деталок** (краулеры обходят id подряд), а не билд. **Метрики важнее догадок** — без дашборда «оптимизировали» бы не то.
- Билд у нас почти бесплатный: **нет `generateStaticParams`** → Next не пререндерит тысячи `[id]`-страниц на сборке. Единственный билд-фетч — `/new` (ISR `revalidate`), 1 запрос. Возиться с ним ради 1 запроса не стоило.

### Data Cache: кэшируй загрузчики деталки
- Загрузчики страницы фильма (`getFilmsById`, `cast`, `box-office`, `facts`, `similar`, `review`, `external`) шли в API **без `next: { revalidate }`** → каждый заход на `/movies/[id]` = ~7 свежих запросов. 2.3K заходов × 7 ≈ 16K обращений/12ч → лимит 500/сутки испаряется.
- Фикс: `next: { revalidate: 60*60*24 }` в каждый `fetch`. Метаданные фильма статичны → сутки. Повторные заходы (особенно ботов на тот же id) идут из **Data Cache на сервере** (общий для всех юзеров), API не трогают.
- Время `revalidate` — баланс: дольше = реже фетчить, но если в кэш попал **сбойный 402**, он залипнет на весь период. Сутки = и экономно, и протухший «не найден» сам обновится за день. ⚠️ `getFilmsById`/`getFilmByIdSafe` бьют в ОДИН эндпоинт (страница + `generateMetadata`) — `revalidate` нужен в обе функции (Request Memoization дедупит их в пределах одного запроса, но не между разными заходами).

### `connection()` ≠ кэш fetch — разные оси (доказано по доке Next)
- Доку читать обязательно (`node_modules/next/dist/docs/.../connection.md`): `connection()` = «рендерить страницу **в рантайме, а не на билде**» (заменяет `unstable_noStore`). Управляет **КОГДА рендерить страницу**.
- `next: { revalidate }` управляет **кэшируется ли fetch между запросами**. Это **другая ось**. Проблему «деталки жгут квоту» лечит именно кэш fetch, а `connection()` тут бесполезен.
- ⚠️ **Полушаг — хуже целого**: убрать `export const revalidate` + сделать `async`, но НЕ добавить `connection()` → страница всё равно пререндерится на билде (ничто не делает её динамической), а ISR-обновление потеряно. Либо целиком (`connection()`), либо не трогать.
- `revalidate` (ISR, page-level) **требует** билд-пререндера (запекает первую версию + фетчит); `connection()` его **отменяет**. Для страницы со списком разница: ISR = статика+1 фетч на билде; dynamic = ноль на билде, кэш на уровне fetch.

### robots.ts — отсечь краулеров от дорогих маршрутов
- App Router: файл `src/app/robots.ts` (в КОРНЕ `app`, группы `(main)`/`(auth)` на URL не влияют) → `export default function robots(): MetadataRoute.Robots`. `import type { MetadataRoute }`.
- `disallow: ["/movies/", "/person/", "/series/"]` — **слеш в конце**: блокирует `/movies/123` (деталку), но оставляет `/movies` (список) открытым. Списки дешёвые/кэшированы → пусть индексируются.
- ⚠️ **Trade-off**: блок деталок = страницы фильмов не попадут в Google (минус SEO). Для учебного проекта на API с лимитом «сберечь квоту» > «SEO»; для прода решали бы кэшем + платным тарифом, а не блоком.
- Кэш fetch гасит повтор на ОДИН id; robots гасит обход МНОГИХ разных id. Нужны обе меры.

### Прочее по квоте
- **Dev и prod не должны делить один лимитированный ключ**: один `API_KEY` в `.env` локально и в Vercel → из ведра 500/сутки черпают и разработка, и билды, и трафик прода (+боты). Разные окружения → разные ключи.
- Проверка квоты ключа: `GET /api/v1/api_keys/{apiKey}` → `dailyQuota: { value: 500, used: N }`. `used > 500` → 402. Свежий ключ имеет `used: 0–3`; если у «нового» ключа `used: 1000+` — он не свежий.
- `.env` Next читает **только при старте** dev-сервера (на лету не подхватывает). `.env.local` перекрывает `.env`.
- ⚠️ **Git-грабля (повтор урока)**: `git add file` пока файл ПУСТОЙ (только создан) → в индекс попадает пустой blob; написанное потом содержимое остаётся в рабочем дереве. `git commit` берёт из индекса пустую версию (и засасывает уже-staged файлы в чужой коммит). Перед коммитом — `git status`, после — `git show HEAD:file`.

### Dropdown: закрытие по клику вне (паттерн оверлей)

- `useEffect + mousedown + ref.contains()` — классический паттерн, но на bleeding-edge стеке (React Compiler) ненадёжен: эффект может не срабатывать из-за мемоизации.
- **Паттерн оверлей** — проще и надёжнее: рендерить `<div className="fixed inset-0 z-40" onClick={() => setIsOpen(false)} />` когда dropdown открыт. Оверлей занимает весь экран под выпадашкой (`z-40`), сама выпадашка поверх (`z-50`). Клик куда угодно вне — попадает на оверлей → закрывает.
- Клик по ссылке внутри выпадашки не закрывает её через оверлей (z-50 > z-40), поэтому нужен явный `onClick={() => setIsOpen(false)}` на каждой ссылке.
- После навигации компонент в хедере остаётся смонтированным с `isOpen: true` — именно поэтому нужен `onClick` на ссылках.

### `useInfiniteQuery` — бесконечная прокрутка

- **Зачем**: держит все загруженные страницы в массиве, умеет подгружать следующую. Обычный `useQuery` — одна страница.
- **4 ключевых поля**:
  - `queryKey` — зависимости (те же что у `useQuery`)
  - `queryFn: ({ pageParam }) => fetchMovies({ page: pageParam, ... })` — получает текущий номер страницы
  - `initialPageParam: 1` — с какой страницы начинать
  - `getNextPageParam: (lastPage, allPages) => allPages.length < lastPage.totalPages ? allPages.length + 1 : undefined` — вернуть `undefined` = страниц больше нет
- **Данные**: `data.pages` — массив всех загруженных страниц. Плоский список: `data?.pages.flatMap(page => page.items) ?? []`
- **Возвращает**: `{ data, fetchNextPage, hasNextPage, isFetchingNextPage }`. Кнопка «Загрузить ещё» — показывать если `hasNextPage`, дизейблить если `isFetchingNextPage`.
- **Тот же мост server action**: клиент не может читать `API_KEY` → `fetchMovies` в `actions.ts` с `"use server"` вызывает существующие загрузчики.

### Дубли из API в `useInfiniteQuery`

- API может вернуть один и тот же фильм на разных страницах → `key={film.kinopoiskId}` не уникален → React ругается.
- Дедуплицировать перед рендером: `films.filter((f, i, self) => self.findIndex(x => x.kinopoiskId === f.kinopoiskId) === i)` — оставляет только первое вхождение каждого id.

### TypeScript: `as unknown as T` и union-ы разных эндпоинтов

- `fetchMovies` возвращает union двух разных форм (`getFilms` → `FilmResponse`, `getFilmsByKeyword` → нормализованный объект). TypeScript не может свести их автоматически.
- Когда типы слишком разные для прямого `as T` — двойное приведение `as unknown as T`. Осознанное «я знаю что делаю» — оба источника реально совместимы, компилятор просто не видит.
- `items: [] as Film[]` — явная аннотация пустого массива в fallback-е, иначе TypeScript выводит `never[]`.

### `IntersectionObserver` — автоподгрузка при скролле

- Браузерное API: следит за тем, попал ли элемент в зону видимости viewport.
- Паттерн: невидимый **элемент-маяк** (`<div ref={sentinelRef} />`) в конце списка. Как только маяк появляется → вызываем `fetchNextPage()`.
- Структура в `useEffect`:
  ```tsx
  const observer = new IntersectionObserver(([entry]) => {
    if (entry.isIntersecting && hasNextPage) fetchNextPage();
  });
  if (sentinelRef.current) observer.observe(sentinelRef.current);
  return () => observer.disconnect(); // cleanup — отписаться
  ```
- Зависимости `[hasNextPage, fetchNextPage]` — пересоздавать обсервер при смене этих значений.
- `if (sentinelRef.current)` — проверка обязательна: при первом рендере ref может быть `null`.
- Кнопка «Загрузить ещё» остаётся как fallback — пользователь может и докрутить и кликнуть, они не мешают друг другу.

### URL-параметр для переключения режима

- `?view=pagination` / `?view=infinite` — хелпер `viewParams(v)` сохраняет остальные параметры (`keyword`, `type`, `genre`), меняет только `view`. Тот же паттерн что `paginationParams`.
- Серверная страница читает `view` из `searchParams`, условно рендерит либо `<InfiniteMovies>` (клиент), либо `<MovieGrid>` + кнопки пагинации (сервер).

### Устойчивость загрузчиков к сетевым сбоям (`try/catch`)

- **`fetch` не бросает при 4xx/5xx** — он возвращает `Response` с плохим статусом. Бросает только при сетевом сбое (`ConnectTimeoutError`, `TypeError: Failed to fetch`). Поэтому `try/catch` нужен именно для сетевых ошибок, а `!response.ok` — для HTTP-ошибок.
- **`!response.ok` заменяет все отдельные проверки статусов**: `status === 404 || status === 402 || status === 400` → просто `!response.ok`. `response.ok` = `status >= 200 && status < 300`.
- **Нужен ли хелпер `safeFetch`?** Нет — каждая функция возвращает разное (`[]`, `null`, `notFound()`). Хелпер не устранил бы дублирование, только добавил новый файл и новый тип. Прямой `try/catch` в каждой функции — проще и читаемее.
- **`throw` внутри `try` — ловушка**: `throw new Error(...)` внутри `try` сразу попадает в `catch` того же блока. IDE предупреждает: «throw of exception caught locally». Решение — `return` вместо `throw`.
- **`notFound()` имеет тип `never`** — TypeScript знает что выполнение после него не продолжится. `return notFound()` — лишний `return`. Достаточно просто `notFound()`.
- **`response` вместо `data`**: `fetch` возвращает `Response` — называть переменную `data` неточно. `data` — для уже распарсенного JSON (`await response.json()`).
- **Изменение сигнатуры функции требует проверки всех мест использования**: `getBoxOffice` раньше возвращал `items[]` напрямую, после рефактора — `BoxOfficeResponse` (`{ total, items }`). Вызывающий код сломался молча (`boxOffice.map is not a function`) — TS не поймал, потому что тип пропа не обновили. Урок: менял сигнатуру — проверь grep по всем вызовам.

### `error.tsx` — границы ошибок (Error Boundaries)

- **Два класса ошибок (по доке Next):**
  - *Expected errors* (ожидаемые) — сбой сети, 404, провал валидации. Моделируются **как значения**: поймал `try/catch`, вернул `[]`/`null`/`notFound()`, рендер продолжается. Это то, что мы уже сделали в загрузчиках.
  - *Uncaught exceptions* (непойманные) — баги: `.map` по `undefined`, чей-то `throw`, падение либы **во время рендера**. Их `try/catch` вокруг fetch не ловит. Для них — `error.tsx`.
- **`error.tsx` = React Error Boundary** вокруг сегмента роута. Обязательно `'use client'` (механизм `getDerivedStateFromError` — классовый клиентский компонент).
- **Что оборачивает:** `page.tsx`, `loading.tsx`, `not-found.tsx` и **вложенные** `layout.tsx`. Что **НЕ** оборачивает — `layout.tsx`/`template.tsx` **своего же** уровня (граница уже отрендерена внутри них). Для ошибок корневого layout — отдельный `global-error.tsx` (со своими `<html>`/`<body>`).
- **Вложенность:** ошибка всплывает до **ближайшей** родительской границы. У нас `[id]/error.tsx` ловит только деталку, `movies/error.tsx` — список + фолбэк для детей без своей границы. Не дублируют, а покрывают разные сегменты.
- **Пропсы:** `{ error: Error & { digest?: string }, unstable_retry: () => void }`. Внутри обычно `useEffect(() => console.error(error), [error])` (на проде — Sentry).
- **`unstable_retry` vs `reset` (версионный нюанс, добавлен в Next 16.2):**
  - `reset()` — просто перерисовать тех же детей теми же данными.
  - `unstable_retry()` — **заново фетчит и рендерит** сегмент. Дока толкает к нему: при *временном* сбое (моргнула сеть, API отдал 500 на секунду) повторный фетч реально восстанавливает. Против **детерминированного** бага (безусловный `throw`) бесполезен — снова та же ошибка, цикл.
- **`digest` и dev-vs-prod message (безопасность!):** серверный компонент имеет доступ к секретам (ключи, строки БД). Если его `message` уедет в браузер — слив. Поэтому:
  - **prod:** настоящий `message` остаётся в серверных логах, на клиент — обобщённое «An error occurred…» + **`error.digest`** (хеш-«номер квитанции»). Тот же хеш в логах → грепаешь и находишь причину. Связь юзер↔лог есть, утечки нет.
  - **dev:** настоящий `message` прокидывается на клиент (удобно дебажить). Отсюда расхождение: при тесте в dev видно текст `throw`, в проде был бы только digest.
  - `digest?` опционально (`?`), т.к. у *клиентских* ошибок его нет — там и так настоящий `message`.
  - **Грабли:** при выводе digest guard'ить надо **`error.digest`**, а не `error`. `error` приходит всегда (граница без него не рендерится) → `{error && ...}` всегда true → при отсутствии хеша покажет `Код ошибки: undefined`. Правильно: `{error.digest ? \`Код: ${error.digest}\` : ""}`. Юзеру показываем digest как «код для поддержки», а сам `error.message` со страницы лучше убрать (для нас есть `console.error` + серверные логи).
- **Как тестировать:** граница молчит, пока что-то не кинет `throw`. Вставить временный `throw new Error("...")` в серверный компонент-ребёнок нужного сегмента (у нас — в `MoviesList`, до `await`). В dev поверх рисуется error-overlay с настоящим сообщением, стилизованный фолбэк — под ним. После проверки `throw` убрать.

### Водопады данных в RSC (`Promise.all`)

- **Водопад (waterfall)** — независимые запросы стоят в очереди из-за последовательных `await`. `await` = знак «стоп»: `const a = await f(); const b = await g();` → `g()` НЕ стартует, пока `f()` не закончил. Если `a` и `b` не зависят друг от друга — это потеря времени на ровном месте.
- **Где было:** деталка фильма `movies/[id]/page.tsx` делала ~6 последовательных `await` (`getFilmsById`, `getCastById`, `getBoxOffice`, `getFilmFacts`, `isFavorite`, `getWatchStatus` + `getSession`). Все берут только `id` → независимы. Цена: сумма всех задержек внешнего API (~6×200мс ≈ 1.2с).
- **Механика фикса:** вызов async-функции возвращает промис **и сразу запускает работу**; `await` — лишь «дождись». Значит: запусти все промисы без `await`, потом `const [a, b, ...] = await Promise.all([...])` — ждёшь только самый медленный (~200мс вместо 1.2с).
- **Тонкости:**
  - `await params` (и `id`) — **первым**, до `Promise.all`: параллелить можно только то, что идёт после получения `id`.
  - Порядок в массиве `Promise.all([...])` = порядок в деструктуризации `[a, b, ...]`. Не перепутать.
  - Вычисляемое из результатов (`directorName = director.find(...)`, `rating = movie.ratingKinopoisk`) — **после** `Promise.all`, в массив не кладётся (это не фетч).
  - `await headers()` лучше вынести строкой выше, а не внутрь массива (иначе чуть тормозит старт; хотя `headers()` локальный → на практике ноль).
- **Грабли (наступил):** `const [...] = Promise.all([...])` **без `await`** — деструктурируешь сам объект-промис, не массив результатов → `movie` undefined → `movie.x` падает. `Promise.all` параллелит, но это всё ещё промис — дождись его.
- **Связь с устойчивостью загрузчиков:** `Promise.all` падает целиком, если хоть один промис reject (иначе — `Promise.allSettled`). Но наши загрузчики обёрнуты в `try/catch` и не бросают (возвращают `[]`/`null`) → reject'а нет → `Promise.all` безопасен. Прошлая работа окупилась.
- **Второй ярус водопада (на потом):** дочерние async-компоненты (`CastSection`, `ReviewsSection`, `SimilarSection`, `StreamChips`) фетчат сами и начинают только ПОСЛЕ `Promise.all` родителя → родитель→дети тоже водопад. Лечится `<Suspense>`-границами: оболочка страницы показывается сразу, секции стримятся независимо.
