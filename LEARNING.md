# Учебный конспект — проект Cinema

Журнал изучения Next.js / React / TypeScript на пет-проекте «кинотеатр».
Цель файла — портативная память: переносится между компьютерами через git.

## Стек
- **Next.js** (App Router; нестандартная сборка — гайды в `node_modules/next/dist/docs/`, читать перед написанием кода)
- **Prisma + PostgreSQL**, **Better Auth**, **Tailwind**, **shadcn/ui**, **Swiper**, **MagicUI**
- Данные о фильмах — **Kinopoisk API Unofficial** (`kinopoiskapiunofficial.tech`), OpenAPI-спека: https://kinopoiskapiunofficial.tech/documentation/api/openapi.json
  - ключ в `API_KEY`, базовый URL в `NEXT_PUBLIC_API_URL`; загрузчики в `src/shared/api/*.ts`, путь `…/api/v2.2/films/{id}/…`

## Команды / окружение
- Prisma — через `bun --bun run prisma ...` (грузит `.env`; `npx` не подхватывает `DATABASE_URL`)
- CLI `tsc` НЕ работает (tsconfig ждёт TS 6.0) → проверка типов только в Webstorm; базовая проверка — `npm run lint`
- ⚠️ **ESLint НЕ ловит**: несуществующие не-JSX идентификаторы (`view`, `emptyStates`), хотя `react/jsx-no-undef` ловит ненайденные JSX-компоненты. Типы краснеют только в Webstorm.
- Dev-сервер обычно запущен — результат виден сразу.

## Что построено
- **Auth**: логин/регистрация, защита `/profile`, восстановление пароля.
- **Избранное / Watchlist**: модели `Favorite`/`Watchlist` (+ enum `WatchStatus`), server actions `toggleFavorite`/`setWatchStatus`, кнопки на странице фильма с **оптимистичным откликом** (`useOptimistic`, клиентские `FavoriteButton`/`WatchButtons`).
- **Профиль**: вкладки (избранное/просмотрено/планы/настройки), `MovieGrid`, счётчики, сортировка (`?sort` по имени/году/рейтингу), переключатель Сетка/Список (клиентский `useState`).
- **Страница фильма**: постер, действия, бокс-офис («Бюджет и сборы»), «Факты и ляпы» (2 случайных факта), актёры (с кнопкой «Показать всех»), режиссёры, съёмочная группа (группировка в Аккордеоне), рецензии, «где смотреть», похожие.
- **Лента `/movies`**: страница-список — серверный `async`-компонент фетчит `getFilms()` и рисует `MovieGrid` (грид). **Пагинация через URL** (`?page=`): кнопки «Назад/Вперёд» на `<Link>`, на краях — `disabled` без ссылки.
- **UI-эффекты**: SparklesText (логотип), AuroraText (заголовки подборок).

## Концепции (главное, что усвоено)

### Server / Client компоненты (RSC)
- Серверный компонент: `async`, фетчит данные, НЕ имеет `useState`/`onClick`/`useEffect`.
- Клиентский: первая строка `"use client"`; только там живут хуки и обработчики событий.
- Паттерн: **сервер фетчит данные → передаёт пропсами → клиент рисует интерактив** (`ProfileMovies`, `CastList`).
- **Граница server→client**: пропсами едут только СЕРИАЛИЗУЕМЫЕ данные. Фильтрация и тяжелая логика (группировка) — на СЕРВЕРЕ; клиент получает готовый чистый массив.

### Трансформация данных и Группировка
- **Метод `.reduce()`**: мощный инструмент для преобразования массива в объект (Record). Позволяет за один проход разложить элементы «по коробкам» (например, по профессиям).
- Синтаксис: `array.reduce((acc, item) => { ... return acc }, {} as T)`. Важно всегда возвращать аккумулятор и инициализировать его правильным типом.
- **Data-driven группировка**: использование внешнего справочника (`PROFESSION_LABELS`) как фильтра и источника названий одновременно.

### Компонентный подход (DRY)
- Вынос повторяющейся верстки в мелкие компоненты (`CastItem`).
- Если компонент не использует хуки, его можно оставить серверным (RSC) для лучшей производительности.
- Передача объекта целиком через проп (`member: Staff`) делает интерфейс чище, но требует внимательной типизации аргументов функции: `function Comp({ member }: Props)`.
- **Переиспользуемый компонент владеет только ОБЩИМ.** То, что различается между местами использования (контейнер/размеры, `sizes`, `loading`), — отдать пропсами или родителю, не зашивать чужую вёрстку. Пример: `PosterImage` рисует только `<Image>` + фолбэк, а `relative`-контейнер с размером/скруглением даёт родитель (карточка грида, строка, страница фильма — у всех разный).
- При выносе логики дублирующий код «худеет»: `Poster` был клиентским (`useState`+`onError`), после выноса в `PosterImage` стал тонкой серверной обёрткой.

### Картинки и фолбэк
- **Фолбэк битой картинки**: `<Image onError={() => setError(true)}>` → при ошибке прячем картинку (`hidden`) и показываем заглушку. `onError` — обработчик события → компонент должен быть клиентским (`"use client"`).
- **`Skeleton`/`animate-pulse` = ВРЕМЕННОЕ ожидание** (грузится). Для «данных нет / не загрузилось НАВСЕГДА» — статичный плейсхолдер (`bg-muted` без пульсации + иконка `ImageOff` по центру). Пульсирующий скелетон на постоянную ошибку обманывает — выглядит как «висит».
- Заглушка поверх `<Image fill>` сама должна быть `absolute inset-0` (картинка `fill` — абсолютная; обычный flow-div схлопнется в 0 высоты).

### Оптимистичный UI (`useOptimistic`)
- **Зачем**: показать ожидаемый результат МГНОВЕННО, не дожидаясь ответа сервера. При ошибке React откатывает состояние сам.
- **Где живёт**: это React-хук → только в клиентском компоненте (`"use client"`). Серверная страница фетчит начальное состояние и отдаёт пропсом — паттерн «сервер фетчит → клиент рисует».
- **Сигнатура**: `const [optimistic, setOptimistic] = useOptimistic(actualState, (current, next) => newState)`. `actualState` — пропс с сервера (источник правды); кнопку рисуем от `optimistic`, не от пропса напрямую.
- **`action` формы становится inline async**: сначала `setOptimistic(...)` (предсказание), потом `await serverAction(...)` (реальная мутация + `revalidatePath`). `.bind(null, ...)` тут НЕ годится — нужно два действия по очереди.
- **Редьюсер обязан зеркалить серверную логику**, иначе UI «дёрнется» при ответе. Булево → `(_, next) => next`; enum-тогл (как watchlist) → `(cur, clk) => cur === clk ? null : clk` (ровно как `if (existing.status === status) delete else upsert` в экшене).
- **Общее состояние — общий хук**: две кнопки watchlist меняют один статус → один `useOptimistic` на обе, не два независимых (иначе рассинхрон подсветки).
- **Когда оправдан**: частые, лёгкие, легко-откатываемые действия (лайк, избранное, чекбокс). НЕ для необратимого/важного (оплата, удаление) — там честно ждать сервер.
- Минус: без JS не работает (серверная `<form action>` работала бы и без JS — progressive enhancement). Для лёгких тоглов компромисс оправдан.

### TypeScript
- **`Partial<Record<Keys, Type>>`**: когда объект содержит только часть ключей из перечисления (например, не все профессии из API нам нужны в справочнике).
- **Дженерики `useOptimistic<State, Action>`**: явно типизировать, когда тип «что показываем» (`WatchStatus | null`) отличается от типа «что передаём в set» (`WatchStatus`).
- **Структурная типизация — проси самый узкий тип, который используешь.** Компонент не должен требовать «жирный» конкретный тип, если читает горстку полей. Описать минимальный тип пропа (`MovieItem`) — и ему подойдут любые объекты с нужными полями (`Film` И `FilmDetail`). Так `MovieGrid` стал переиспользуемым для каталога и профиля сразу.
- **`?` vs `| null` — разное!** `field?: T` — ключа может вообще не быть. `field: T | null` — ключ обязан быть, но значение может быть `null`. Чтобы тип подошёл объекту, где поля НЕТ совсем (`Film` без `filmLength`), нужен именно `?`.

### React / JSX
- Правая часть `&&` = ОДИН узел; несколько детей → один `<>...</>` (Fragment); JS-выражение в разметке → в `{ }`.
- `key` — на самом внешнем элементе, который возвращает `.map`. Для key + несколько соседей: `<Fragment key={i}>` (короткий `<>` не принимает key).
- Сепаратор между элементами: `{i < items.length - 1 && <Separator/>}`.
- Тернарник `активна ? "secondary" : "ghost"` — подсветка всегда в ПЕРВОЙ ветке (легко перепутать).
- `dangerouslySetInnerHTML={{ __html }}` — для рендера сырого HTML (React по умолчанию экранирует `{text}`); у такого элемента не может быть детей; контейнер — `<div>` (разрешает блочное).
- Data-driven рендеринг: справочник-объект/массив (`emptyStates`, `sorts`, `typeLabels`) вместо копипаста разметки; `Record<...>` + `.map`.

### Next.js: загрузка и стриминг (`loading.tsx`)
- **`loading.tsx`** рядом с `page.tsx` — конвенция: Next САМ оборачивает страницу в `<Suspense fallback={<Loading/>}>`. Пока `async`-страница ждёт данные — виден фолбэк, потом контент подменяется (стриминг). Ручной `<Suspense>` писать не нужно.
- Серверный компонент, БЕЗ пропсов, **default export**. Оборачивает `page.tsx`, но не `layout.tsx` того же сегмента.
- **Скелетон повторяет сетку контента** (те же `grid-cols`, `gap`, `aspect-2/3`) — иначе вёрстка прыгнет при подмене.
- N заглушек без данных: `[...Array(20)].map((_, i) => <Skeleton key={i} .../>)`. Готовый `Skeleton` = `animate-pulse rounded-xl bg-muted`.
- Проверка стриминга в HTML: в одном ответе есть И фолбэк-скелетон, И реальные карточки. Увидеть глазами — троттлинг Slow 3G в DevTools.

### Next.js: URL как состояние (`searchParams`)
- **`searchParams` — это `Promise`** (Next 16): `async function Page({ searchParams }: { searchParams: Promise<{ page?: string }> })` → `const { page } = await searchParams`. Причина: статику Next рендерит наперёд, а query известно только в момент запроса; `await` осознанно переводит кусок в динамический рендер. Так же устроен `params` для `[id]`.
- **`params` ≠ `searchParams`**: `params` — сегменты пути (`[id]`), `searchParams` — query после `?`. У страницы без `[...]` сегмента `params` пустой.
- **Из URL всё приходит строкой** (`?page=3` → `"3"`). В число: `Number(page) || 1`. Порядок важен — `|| 1` ловит `NaN` ПОСЛЕ `Number` (и для пустого, и для мусора `?page=abc`); `Number(page ?? 1)` мусор не ловит (`??` срабатывает только на null/undefined, а `"abc"` для него «есть»).
- **URL-пагинация = `<Link href="?page=...">`** (не кнопки с onClick). Клик меняет `searchParams` → страница перефетчивается. Без клиентского JS.
- **«Выглядит disabled» ≠ «заблокировано»**: `disabled` на кнопке внутри `<a>` — косметика, ссылка всё равно живая. Чтобы реально не листало на краю — условный рендер: `isFirst ? <Button disabled> : <Button asChild><Link/></Button>` (на краю ссылки нет совсем).
- **`asChild`** (shadcn/Radix `Slot`): `Button` не рисует свой `<button>`, а «переливает» стили на дочерний `<Link>` → один валидный `<a class="кнопка">` вместо невалидной вложенности `<a><button>`. Приём для любого «кнопочного вида на ссылке».

### CSS / Tailwind
- `gap`/`flex` ставится на РОДИТЕЛЯ (разносит детей), не на каждого ребёнка.
- `<Image fill>` требует родителя с заданным размером (высотой): `aspect-2/3` или `h-28 w-20`.
- Скругление картинки: `rounded-*` + `overflow-hidden` на контейнере (один `border-radius` не обрезает содержимое).
- `toLocaleString("ru-RU")` для разрядов чисел (`25000000` → `25 000 000`).

### Архитектурные привычки
- Читалки данных — в `src/lib/*.ts` (без `"use server"`); мутации — `actions.ts` рядом со страницей.
- Статика, нужная в нескольких файлах → `shared/constants/*`.
- При выносе компонента переезжают и его данные (импортом); на старом месте чистить вызовы И мёртвый код (полу-миграция = висячие ссылки с обеих сторон).
- Не хардкодить значение из данных в условие, если API даёт нужное поле (был `i.symbol === "$" ? …` — терял не-долларовые валюты).
